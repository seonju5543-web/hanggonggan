## 서비스 개요
**한대장(한국 대학교 장학금)**: 대학생이 프로필 1회 등록으로 교내·교외 장학금을 자동 매칭받고,
증명서류 준비와 **공고 원본 양식 작성**까지 앱에서 끝내는 모바일 PWA.
- 배포 주소: https://seonju5543-web.github.io/hanggonggan/ (main에 push하면 자동 배포)
- 학생 영구 무료. 수익화 전략은 `BUSINESS.md`.

## 운영 원칙 (개발자와 합의됨 — 변경 금지)
1. **정직한 신청 상태**: 실제 접수가 일어나지 않는 것을 "신청 완료"로 표기하지 않는다.
   현재 모델 = "신청 준비 완료" + 최종 제출처 안내. 이메일 접수 공고만 실제 전송 가능.
2. **장학금 정식 등록(매칭+양식 지원)은 개발자 컨펌 후에만** — 단, **2026-07-15 개발자 지시로 자동 등록
   예외가 추가됨**: 수집 로봇이 보수적 규칙(개별 실공고·미등록·마감 전·대출/행사/고용/대학원 제외)을
   전부 통과한 공고만 `collector/auto-register.mjs`로 자동 등록하고, 앱에 **'자동 등록 · 검수 전' 배지**를
   붙여 정직하게 표시하며, 리포트 이슈에 선조치후보고한다. 끄기: `collector/auto-register-config.json`
   `enabled:false` / 잘못 등록분 제거: 같은 파일 `blockIds`에 id 추가. 애매한 건(타교 접수분·미래 사업 등)은
   자동 등록하지 않고 '컨펌 대기'로 리포트에만 올린다 — 이 컨펌은 여전히 채팅에서 개발자가 직접 지시한다.
   실시간 공고 피드(제목+링크)는 컨펌 없이 자동 게재 — 이건 합의된 예외.
3. **선조치후보고**: 방향 결정이 필요하면 최적안으로 먼저 조치하고 "어떤 상황에서 어떤 조치"를 보고.
4. **양식의 정의**: 공고에 첨부된 실제 신청서 파일(HWP 등)과 **동일한 구조**의 문서를 생성해야 한다.
   자유 서식 텍스트로 대체하면 안 됨. 신규 양식은 스키마화 후 `data/forms.json`에 등록.
5. **양식 확보 절차 (개발자 컨펌 이후 — 정식 등록의 기본 절차)**: 정식 등록할 때 첨부에 신청서
   양식 파일이 있으면 **양식 등록까지가 한 세트**다.
   ① `collector/run-deepfetch.txt`에 `targets: 공고키워드1,키워드2` 줄을 적어 기본 브랜치에 push(push-to-run)
   → 첨부 원본 + HWP 미리보기 텍스트(.hwp.txt)가 `collector/extracted/`에 자동 커밋됨
   ② 원본 항목·문구·체크옵션 그대로 `data/forms.json`에 스키마화, registered.json 항목에 `formId` 연결
   ③ `verify/verify-registered.js`·`verify-forms-data.js`로 검증 후 배포 (앱 파일 무변경 — 자동 반영).
   제출 방식이 양식 제출이 아니거나(포털 입력형 등) 양식 확인에 문제가 있으면,
   **모든 수단(공고 상세, 첨부, 주관 재단 홈페이지, fetch-page 원격 열람, 정찰 도구)을 동원해 찾아본 뒤**
   그래도 못 찾을 때만 개발자에게 필요한 정보를 정리해 문의한다.
   채울 신청서 양식이 아예 없는 공고는 registered.json 항목에 `noForm: "사유"`를 기재한다(감사 통과 조건).
6. **가짜 공지 금지 (2026-07-05 개발자 지시)**: 임의로 만든 장학금 공지를 사용자에게 보여주지 않는다.
   실존 상시 제도(국가장학금 등)는 유지하되 임의 마감일 없이 '상시 제도' 배지 + 재단 공지 확인으로 표기.
7. **소급 적용 원칙 (2026-07-05 개발자 지시)**: 엔진·정책이 업데이트되면 새 데이터뿐 아니라
   **기존 데이터 전체에 적용**해야 한다. 강제 장치: `node verify/audit-data.js` — 정식 등록·양식
   전수를 현재 기준으로 감사(양식 첨부 있는데 formId 없으면 경고). 페이스리프트·엔진 변경 후 필수 실행.
8-1. **추론 금지 · 원문 발췌 원칙 (2026-07-15 개발자 지시)**: 공고의 신청 방법·요구 서류를 앱이
   '추론'해 제시하면 안 된다. 확보된 공고 전문에서 **원문 문장을 그대로 발췌**(`collector/
   extract-excerpts.mjs`, 수집 때마다 자동 갱신 — registered.json `excerpts`)해 "공고 원문 안내 ·
   원문 그대로" 블록으로만 표시하고, 발췌가 없으면 지어내지 않고 '원문 보기' 링크만 준다.
   금액 미확인 공고는 amountValue 0 — 합계에 지어낸 숫자를 섞지 않고 '금액 미확인 n건 제외'로 표시.
8-2. **앱 제작 지원문서 한정 원칙 (2026-07-15 개발자 지시)**: 앱이 직접 만든 지원문서는 **자유 형식
   제출이 원문으로 확인된 공고(registered.json `prepDoc: true`)에만** 제공한다(예: 불광 — 지정 양식
   없음 확정·이메일 제출). 제출할 수 없는 공고에 '준비용 문서'를 붙이는 것은 사용자 혼란·신뢰 훼손이라
   금지. 등록·검수 때 원문에서 자유 형식 제출이 확인되면 prepDoc을 켠다.
8. **진척도 기록 방식 (2026-07-05 합의)**: 공식 제출·발표 결과는 앱이 학교 전산을 볼 수 없으므로
   **사용자가 직접 기록**한다(제출했어요/선정·미선정 버튼). '심사' 단계만 객관적 사실
   (제출 기록 + 접수 마감 경과)로 자동 표시. 장학팀 파일럿이 성사되면 자동화로 전환 가능.

## 중요한 기술 사실 (재발견에 시간 쓰지 말 것)
- 이 저장소 **기본 브랜치는 `claude/nice-heisenberg-WESq5`** (예약 워크플로는 여기 있어야 작동).
  **Pages 배포는 main 브랜치 소스** — 배포하려면 main에 push. 작업 브랜치는 `claude/korean-scholarship-app-8posyu`.
  세 브랜치를 모두 같은 내용으로 push하는 것이 관례 (main push 전 `git fetch origin main` — 봇이 커밋함).
- Claude 실행 샌드박스는 학교 사이트·github.io 접속 차단 → 외부 확인은 GitHub Actions(로봇/정찰/fetch-page)로.
- **GitHub MCP 연결이 없는 세션에서도 워크플로 실행 가능**: deep-fetch/browser-collect는 `collector/run-deepfetch.txt` / `collector/run-browser-collect.txt`를 수정해 기본 브랜치에 push하면 자동 실행된다(push-to-run). 결과는 봇이 기본 브랜치에 커밋 → `git fetch` 후 merge로 읽는다.
- **자동 업데이트 체계 (2026-07-05 구축)**: ① 데이터(`data/*.json` — 공고·정식등록·양식)는 서비스워커가 네트워크 우선으로 받고, 앱을 열 때마다 + 화면 복귀 때마다(5분 간격 제한) 다시 읽는다 → **재설치 없이 자동 반영**. ② 앱 코드(html/js/css)를 바꾸면 CACHE 버전(sw.js, 현재 v9)을 올릴 것 — 새 서비스워커가 활성화되면 화면이 자동으로 한 번 새로고침돼 즉시 적용된다(app.js controllerchange). **장학금·양식 추가는 데이터 파일만 바꾸므로 CACHE 버전 인상 불필요.**
- **보안 체계 (2026-07-05 점검 완료)**: index.html에 CSP 메타(외부 스크립트·임의 접속 차단 — 수집 데이터가 오염돼도 코드 실행 불가) + 리퍼러 미전송. 모든 장학금 데이터 렌더링에 esc() 적용. 개인정보(프로필·서류·양식 답변)는 localStorage/IndexedDB **기기 안에만** 저장, 외부 전송 코드 없음(사용자가 직접 공유·메일 열기 시에만 나감). mail-worker는 Origin 검사+수신 도메인 허용목록(ac.kr/or.kr/go.kr/re.kr)+크기 제한으로 스팸 릴레이 차단. 워크플로 입력은 환경변수로만 셸에 전달.
- **새 양식 등록은 `data/forms.json`에만 추가**하면 된다(forms.js·app.js 무변경). registered.json 항목에 `formId`를 연결하면 신청 플로우에 양식 작성이 자동으로 붙는다. 이 경로의 검증 드라이버: `verify/verify-forms-data.js`.
- **수집 로봇이 새 공고 0건이면 새 리포트 이슈를 만들지 않고** 최근 리포트 이슈에 "정상 실행 — 0건" 코멘트만 남긴다(2026-07-05 추가).
- **예약 실행 혼잡 회피 정책 (2026-07-06)**: GitHub는 UTC 정각·자정 부근 예약을 지연·생략한다(7/5 +3h44m 지연, 7/6 완전 누락 발생). 그래서 수집은 홀수 분 하루 2회 예약 — 일반 07:41/11:41 KST, 브라우저 08:07/12:07 KST. 중복 실행은 seen.json이 걸러 무해. 일반 수집도 push-to-run 지원(`collector/run-collect.txt`).
- iOS Safari는 datalist 미지원(커스텀 자동추천 구현됨), 전역 appearance:none이 체크박스 지움(커스텀 체크 구현됨).
- 검증: **`verify/` 폴더에 Playwright 드라이버 커밋되어 있음** (`verify/README.md` 절차대로 실행 — drive.js 전체 회귀, verify-registered.js 정식등록·조병두 양식, verify-forms-data.js 데이터 주도 양식 반영). 한글 파일명 setInputFiles 실패함(ASCII 사용).
- HWP 원본 해석: `olefile`로 열어 `PrvText` 스트림을 UTF-16LE로 읽으면 양식 전문이 나온다 (조병두 스키마화에 사용한 방법).
- 광운·상명 게시판은 본문을 자바스크립트로 그려서 일반 fetch에는 메뉴만 보임 → 1차 수집 때 저장된 deadlineHint를 활용하거나 브라우저형 로봇 사용.
